# 视频会议管理系统 - 架构与模块设计说明书

## 一、 技术栈选型 (面向AI生成优化的组合)

为了让AI能生成最高质量、最现代化的代码，推荐以下成熟且生态良好的技术栈：

- **后端 (Backend)**: Python + **FastAPI** (支持异步和WebSocket，非常适合实时通信场景)
- **前端 (Frontend)**: **Vue 3** + Vite + Element Plus (符合现代前端规范，组件库丰富，便于快速构建后台界面)
- **数据库 (Database)**: **MySQL** 或 **SQLite** (初期开发推荐SQLite，免配置，后续无缝迁移MySQL) + SQLAlchemy (ORM)
- **实时通信与视频流**:
  - **信令服务器 & 实时状态**: FastAPI **WebSockets** (用于聊天、表决结果实时同步)
  - **音视频核心**: **WebRTC** (端到端音视频通信与桌面共享)。*注：为降低AI生成难度，初期可采用Mesh网络拓扑的纯WebRTC P2P方案，或集成开源的LiveKit SDK。*
  - **直播流**: 使用WebRTC广播模式，或通过后端生成虚拟直播间。

------

## 二、 系统架构设计

系统整体分为四层架构：

1. **表现层 (Vue 3)**: 负责UI渲染、用户交互、音视频流播放展示。
2. **网关/路由层 (FastAPI API Router)**: 处理HTTP请求（RESTful API）和WebSocket长连接。
3. **业务逻辑层 (Python Services)**: 处理会议预约、权限校验、表决统计等核心逻辑。
4. **数据层 (SQLAlchemy + DB)**: 持久化存储用户、会议、表决、录像回放等数据。

------

## 三、 功能模块详细设计 (给AI的开发指令)

请指导AI按照以下四个核心模块逐一进行开发：

### 模块一：视频会议模块 (Video Conference)

**目标**：实现视频会议的预约、回放、通信、桌面共享。

- **后端开发需求**:
  - 设计 `Meeting` 实体表（包含字段：主题、开始时间、结束时间、状态、创建者ID、录像URL）。
  - 提供会议的 CRUD (创建、查询、修改、删除) REST API。
  - 提供基于 WebSocket 的信令服务，用于客户端交换 WebRTC 的 SDP 和 ICE 候选信息，建立音视频连接。
- **前端开发需求**:
  - **预约界面**: 表单填写会议主题、时间，调用创建API。
  - **会议室界面**:
    - 集成 `navigator.mediaDevices.getUserMedia` 获取本地摄像头和麦克风。
    - 集成 `navigator.mediaDevices.getDisplayMedia` 实现**桌面共享**功能。
    - 使用 `<video>` 标签动态渲染参会人员的画面。
  - **回放列表**: 展示已结束会议的录像视频列表，点击可调用 `<video>` 播放器进行回看。

### 模块二：直播模块 (Live Streaming)

**目标**：实现单机直播功能及直播回看（暂不考虑万人并发）。

- **后端开发需求**:
  - 设计 `LiveStream` 实体表（包含直播标题、主播ID、推流地址/房间号、回放URL）。
  - 提供开启直播、结束直播的 API 接口。
- **前端开发需求**:
  - **主播端**: 复用视频会议的音视频采集逻辑，将本地流标记为主播流，并广播给所有订阅者。
  - **观众端**: 以“仅拉流 (Recvonly)”模式加入 WebRTC 房间，不发送本地音视频，仅观看主播画面（或播放 HLS 格式流）。
  - **回看功能**: 提供直播历史列表，支持点击播放录制的直播视频文件。

### 模块三：在线表决模块 (Online Voting)

**目标**：实现在线表决、同步结果并实时展示。

- **后端开发需求**:
  - 设计 `Vote` 表（会议ID、表决主题、选项）和 `VoteRecord` 表（用户ID、表决ID、所选项）。
  - 提供发起表决的 API 接口。
  - **核心逻辑**: 接收到用户的表决提交后，立即写入数据库，并通过 WebSocket 广播最新的统计数据给该会议室内的所有连接客户端。
- **前端开发需求**:
  - **主持人端**: 在会议室面板中点击“发起表决”，填写问题和选项（如：赞成、反对、弃权）。
  - **参与人员端**: 接收到 WebSocket 消息后，弹出表决弹窗。点击提交后，按钮置灰。
  - **实时展示**: 会议界面侧边栏使用图表（可引入 ECharts）或进度条，实时监听 WebSocket 消息并更新各选项的投票人数和比例。

### 模块四：权限设置模块 (Permission & Role Settings)

**目标**：根据账号不同创建并后台设置不同的视频会议室。

- **后端开发需求**:
  - 设计 `User` 表（包含字段：用户名、密码哈希、角色）和 `Role` 字典（如：Admin, Host, User）。
  - 实现基于 JWT (JSON Web Token) 的鉴权机制。
  - **权限拦截**:
    - 只有 Admin 角色可以进入“后台管理系统”创建特殊的会议室/分配账号。
    - 只有 Host 角色可以“发起会议/直播”。
    - 普通 User 角色仅能“加入会议/观看直播”。
- **前端开发需求**:
  - **登录注册页面**: 获取并存储 JWT Token（如放在 localStorage 中）。
  - **动态路由 (Vue Router)**: 根据用户登录后返回的角色信息，动态渲染左侧导航菜单（管理员可见后台设置，普通人只可见会议列表）。
  - **后台管理页面**: 提供用户管理、会议室模板分配的 CRUD 界面。

## 四、 数据库核心模型设计参考 (Database Schema)

可以把这段直接给AI，让它生成 SQLAlchemy 模型：

Python

```
# 核心实体关系
User (id, username, password_hash, role) # role: admin, host, normal
Meeting (id, title, start_time, end_time, host_id[FK:User], status, record_url)
LiveStream (id, title, host_id[FK:User], start_time, record_url)
Vote (id, meeting_id[FK:Meeting], topic)
VoteOption (id, vote_id[FK:Vote], content)
VoteRecord (id, vote_id[FK:Vote], user_id[FK:User], option_id[FK:VoteOption])
```

2. 第二步（权限与基础路由）：

3. 第三步（视频与直播核心）：
   - “根据【模块一】和【模块二】，使用 FastAPI 的 WebSocket 实现 WebRTC 的信令交换。在前端实现创建会议室、调用本地摄像头/麦克风、获取桌面共享流的功能。”
4. 第四步（在线表决）：
   - “根据【模块三】，在刚才的会议室界面中加入在线表决功能，后端新增表决相关的表和接口，必须通过 WebSocket 实时把投票结果推送给前端进行展示。”
