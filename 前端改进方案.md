Context

     用户这次的需求已经变更为“前端页面与布局的视觉效果优化”，与上一次围绕会议室媒体开关和投票逻辑修复 
     的任务不同，因此本次计划需要整体重写。当前前端基于 Vue 3 + Element
     Plus，已经有一定的产品化视觉基础，但样式分散在 global.css、MainLayout.vue、AdminLayout.vue       
     以及多个页面/组件的 scoped CSS 中；同时主站与后台、登录与注册、列表页与房间页之间存在较多重复样式
     与视觉漂移。此次优化的目标是：在不改动业务流程的前提下，统一视觉语言、提升页面层级感与空间节奏， 
     并让主要页面在桌面端与中小屏幕上都更精致、更一致。

     Recommended approach

     1. 先从全局视觉基线入手，集中优化 frontend/src/styles/global.css：统一页面背景、基础字体层级、卡 
     片圆角、阴影、边框、按钮/输入框/对话框/Table 的 Element Plus
     覆盖样式，建立一套更稳定的产品化视觉基调。
     2. 统一应用壳层布局，重点调整 frontend/src/views/MainLayout.vue 与
     frontend/src/views/admin/AdminLayout.vue：收敛侧边栏、顶部栏、内容区
     gutter、最大宽度、信息卡片与导航激活态的视觉表达，减少前后台两套布局的风格漂移。
     3. 优化共享组件作为全站视觉骨架：优先处理 frontend/src/components/layout/PageHeader.vue、frontend
     /src/components/ui/SummaryCard.vue、frontend/src/components/ui/StatusTag.vue、frontend/src/compon
     ents/ui/EmptyState.vue，统一标题区层次、统计卡密度、状态标签语义色、空状态表现和操作区节奏。     
     4. 统一认证页观感，围绕 frontend/src/components/auth/AuthShell.vue 收敛登录/注册页的表单容器、品 
     牌区、辅助文案、按钮主次层级与留白，让入口页更有品牌感而不是工具页风格。
     5. 在共享样式稳定后，再批量微调高频页面视图：优先覆盖
     frontend/src/views/meeting/MeetingListView.vue、frontend/src/views/live/LiveListView.vue、fronten
     d/src/views/admin/UserManageView.vue、frontend/src/views/admin/RoomTemplateView.vue，统一“页头 + 
     概览卡 + 内容卡/表格”模式的布局节奏与视觉层级。
     6. 最后处理实时场景页面，优化
     frontend/src/views/meeting/MeetingRoomView.vue、frontend/src/views/live/LiveRoomView.vue
     及其相关组件 frontend/src/components/media/MediaTile.vue、frontend/src/components/meeting/VotePan
     el.vue，增强房间页的信息分区、按钮主次关系、媒体卡片质感与侧边面板可读性。
     7. 收尾阶段做响应式与状态回归检查：确认 hover / active / focus / disabled / empty / loading      
     等状态风格一致，检查 1100px、900px、720px 等已有断点下是否存在溢出、拥挤或层级失衡问题。

     Verification

     Manual visual review

     - 对比优化前后主站壳层、后台壳层、登录页、注册页、会议列表页、直播列表页、会议室页、直播间页、用 
     户管理页、会议室模板页的整体视觉一致性。
     - 检查页头标题层级、概览卡密度、内容卡/表格容器、按钮主次关系、空状态与状态标签是否形成统一风格。
     -
     检查桌面宽屏、普通笔记本、平板和窄屏下的布局：是否有横向溢出、控件换行异常、内容过挤或留白失衡。 
     - 检查 Element Plus 组件在自定义样式下是否仍保持一致：按钮、输入框、表单、弹窗、表格、进度条、菜 
     单等不能出现视觉割裂。
     - 检查交互状态：hover、active、focus-visible、disabled、loading、empty、error
     等是否都有清晰反馈且风格统一。

     Regression checks

     - 所有已有路由页面应可正常进入，页面结构不应因纯样式优化导致错位或功能区缺失。
     - 长标题、长用户名、长房间名、状态文案较长时，布局不应破坏主结构。
     - 会议室与直播间等密集页面在多卡片、多流媒体情况下仍需保持可读性。

     Optional implementation checks

     - 若项目有可用的前端构建/预览流程，执行一次构建或本地预览检查，确认样式修改未引入编译错误。      
     - 如有截图对比或视觉回归手段，可优先对主壳层、列表页、认证页、房间页做前后对照。
